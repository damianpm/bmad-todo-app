## 1. Executive summary

A minimal, single-user, full-stack Todo application that lets a person capture, view, complete, and delete personal tasks with zero onboarding. The product's defining property is **disciplined restraint**: it ships only the four CRUD verbs and the UI states required to make them feel polished, and explicitly excludes accounts, collaboration, priorities, deadlines, and notifications. Success is measured by whether a first-time user can complete every core action without guidance and whether their data survives refreshes and restarts.

## 2. Problem & opportunity

Most todo products fail in one of two directions: they bury the basic capture/check-off loop under feature creep (projects, labels, recurrence, calendars), or they ship as throwaway demos that lose data on refresh. There is value in a deliberately small reference implementation that does the basic loop *well* — fast, durable, accessible, containerized — and provides a clean foundation for any of those features to be added later without rework.

**Why now:** The exercise also serves as a vehicle to validate an AI-driven, BMAD-structured development workflow end-to-end. The deliverable is both the app and the process documentation around how AI agents helped produce it.

## 3. Target user

**Primary user:** an individual managing a personal task list on their own device(s).

**Implicit secondary user:** the future engineer extending this codebase. Architecture decisions must keep the door open for `userId`, multi-tenancy, and feature additions without contortions.

**Out of audience for v1:** teams, multi-device sync via accounts, organizations, anyone who needs sharing or assignment.

## 4. Vision & value proposition

> "Open the app, see your tasks, capture a new one in one keystroke, check it off, move on. Refresh — it's still there. Open it on your phone — it works."

The user gains a frictionless, durable inbox for personal tasks. The team gains a small, fully-tested, container-deployable codebase that demonstrates each layer (UI, API, DB, tests, containerization) in its simplest correct form.

## 5. Scope (v1)

**In scope:**
- Create a todo (short text, max ~500 chars)
- View all todos (newest first or stable creation order — to be locked in PRD)
- Toggle a todo between "open" and "completed"
- Delete a todo
- Visually distinguish completed vs. active items
- Empty state, loading state, error state UIs
- Responsive layout (desktop and mobile)
- Optimistic UI updates with rollback on API failure
- Persistence across page refreshes, server restarts, container rebuilds
- Health endpoint for container orchestration
- Containerized deployment via `docker compose up`
- Unit, integration, and E2E test coverage with ≥70 % line coverage
- WCAG 2.1 AA accessibility (zero critical violations)

**Out of scope (v1) — explicitly:**
- Authentication / accounts / login
- Multi-user, sharing, assignment
- Priorities, due dates, reminders, notifications
- Tags, projects, labels, search
- Editing the text of an existing todo (flag for PRD confirmation — common ask, but not in source PRD)
- Drag-to-reorder
- Offline mode / PWA / push
- Real-time sync between tabs
- Export / import
- Internationalization beyond English

## 6. Success criteria

## 7. Constraints

**Technical (locked by stakeholder, do not re-evaluate):**
- Frontend: React 19 + Vite + TypeScript
- Backend: Node 24 LTS + Fastify + TypeScript
- Persistence: PostgreSQL 16 in a sidecar container
- ORM: Drizzle (proposed; Architect to confirm via ADR)
- Test stack: Vitest (unit + integration), Playwright (E2E), axe-core (a11y)
- Container orchestration: Docker Compose

**Process:**
- BMAD-METHOD drives planning artifacts (this brief, PRD, architecture, epics, stories).
- AI agent usage is logged in `docs/ai-integration-log.md` throughout.
- Each phase produces persistent, reviewable artifacts before code is written.

**Architectural guardrails:**
- Data model and API shape must permit a `userId` column / scope without breaking changes (no global singletons, no implicit "the user").
- No authentication code paths in v1, but the request lifecycle should have an obvious place to insert auth middleware later.

## 8. Key risks

| Risk | Likelihood | Impact | Mitigation |
|---|---|---|---|
| Scope creep during build (someone adds priorities or due dates "while we're here") | Medium | High — kills the "minimal" thesis | This brief and the PRD enumerate exclusions explicitly; Architect doc will reject ADRs that pull in excluded features. |
| Optimistic-UI bugs leak inconsistent state on failure | Medium | Medium | E2E "error state" test covers API-down rollback; integration tests cover server-side validation rejections. |
| Container DB volume not mounted → data loss across `docker compose down -v` | Low | High in demo | Document the named-volume convention in architecture.md and reproduce in an E2E "data persists across restart" test. |
| Drizzle ORM tooling friction (migrations, native deps in container) | Low | Medium | Captured as ADR in architecture phase; raw `pg` is a fallback. |
| Accessibility violations discovered late | Medium | Medium | axe-core wired into Playwright from the start (Story E1 acceptance). |
| BMAD workflow tooling friction inside Claude Code (skills not auto-registered mid-session) | Realised | Low | Execute skill instructions inline; logged in AI integration log as a Day-1 finding. |

## 9. Dependencies & assumptions

- Single developer / single workstream — no parallel team coordination needed.
